# LT大阪 発表メモ

## 今回の本命テーマ
AIに実装を任せるほど、人間の学びは途中で抜け落ちやすい。  
残すべきなのは完成コードだけではなく、詰まり、失敗、判断の履歴。  
それらを週報という形式で再利用可能な知識に変える発表を軸にする。

---

## 発表コンセプト

### 問題意識
- AIに実装を任せること自体は、これからますます普通になる
- ただし、何も考えずに使うと、人間側に試行錯誤の知識が残りにくい
- 完成物だけ見ても、なぜ詰まったか、何を見落としたか、どこで判断したかは埋もれやすい

### 仮説
- 市場価値を点数化するより、試行錯誤を知識として再利用できる状態を作る方が健全
- AIは人を評価する役割ではなく、学びの再利用を支援する役割に置く
- 失敗を単発イベントで終わらせず、次回の行動に変換できれば価値がある

### 発表で言いたいこと
AIに実装を任せる時代ほど、人間は途中の詰まりや失敗を知識として残す必要がある。

---

## タイトル候補
推奨順はこの順。

1. **AI開発で埋もれる学びをどう残すか**
2. **試行錯誤を知識に変える週報AI**
3. **AIに書かせたコードより、残したいもの**
4. **失敗を資産に変える週報AIを考えてみた**
5. **実装はAIに任せても、学びは人間に残したい**

---

## プロダクト案

### 名前
**試行錯誤を知識に変える週報AI**

### 何をするものか
GitHub PR を中心に、今週の試行錯誤を振り返り可能な形に変換する。  
主役は「完成したコード」ではなく、レビューで指摘されたこと、修正したこと、繰り返している詰まり。

### 対象
- 個人開発者
- AI支援で実装する機会が増えているエンジニア
- 実装は進むが、自分に知識が残っている感覚が薄い人

### 入力
- GitHub PR のレビューコメント
- PR の修正履歴
- 必要に応じて commit message や Issue を将来拡張で追加

### 出力
- 今週の学び
- 繰り返している詰まり
- 次回の自分へのメモ

### ポジション
- 人を採点しない
- 能力を断定しない
- 失敗を笑いものにしない
- 学びの再利用を助ける
- 週報は提出物ではなく、自分の知識ベース

---

## PRを入口にする理由
- 入力ソースが明確で、5分LTでも説明しやすい
- レビューコメントには、自分では気づかなかった視点が残りやすい
- 修正履歴と組み合わせると、「何を学んだか」を比較的自然に取り出せる
- 将来的に Issue や commit へ広げる入口としても妥当

---

## デモ方針

### 5分LTで見せる範囲
- 1つのPRを入力
- レビュー指摘と修正履歴を読む
- 学び3点と次回の注意点を返す
- それを「今週の学び」として週報風にまとめる

### デモで強調すること
- AIが正解を出すのではなく、埋もれる学びを拾い上げる
- レビューをその場限りで終わらせず、次回に使える形にする
- 完成コードの評価ではなく、試行錯誤の再利用が主題

### デモの現実解
- ライブで全自動処理を見せるより、サンプルPRデータ前提でもよい
- リスクの高い部分は事前に用意し、発表では入力と出力の変化に集中する

---

## 5分LTの流れ案

### 1. 導入
- AIに実装を任せるのは便利
- でも、それだけだと人間側に何が残るのかが気になった

### 2. 問題提起
- コードは残るが、詰まりや判断の履歴は残りにくい
- そこに学びの源泉があるのに、日々の開発では流れていく

### 3. 仮説
- 失敗や試行錯誤を、週報という形で知識化できれば再利用できるのではないか

### 4. 実装
- GitHub PR を入力にして、レビューから学びを抽出する週報AIを作ってみた

### 5. まとめ
- 市場価値を点数化するより、学びを残せる状態の方が健全
- AI時代に残したいのは、完成物だけでなく途中で得た知見

---

## 伝え方の注意
- 「AIが評価する」ではなく「AIが整理を手伝う」と表現する
- 「市場価値アップ」ではなく「学びの再利用」から入る
- ネガティブな失敗談の消費ではなく、次回に活きる知識化として話す
- 会の方向性に合わせて、煽りよりも内省を前に出す
