# 実装計画レビュー

対象: `IMPLEMENTATION_PLAN.md`  
目的: 実装前に、強い点、曖昧な点、先に詰めるべき点を構造化して残す。

---

## 総評
- 全体の方向性は良い
- `何を作るか` と `なぜこの技術を選ぶか` は十分に整理されている
- 一方で、実装着手前に詰めるべきなのは `MVPの細さ` `学び抽出の評価方法` `GitHub App導入の現実的な運用` の3点

---

## 再評価メモ
- LT時点で完成している必要はない
- 最悪、個人ツールとして動いていれば十分
- この前提に立つと、スタック規模とデモリスクに関する懸念はかなり弱まる
- そのうえで残る主な論点は `認証設計` と `Phase 5 の位置づけ` である

## 良い点

### R-001
- 優先度: high
- 論点: 問題設定
- コメント: `AIに実装を任せるほど、人間の学びが残りにくい` という問題設定が明確で、技術選定や機能設計がこの問題にぶら下がっている
- 残す価値: プロダクトの芯として維持すべき

### R-002
- 優先度: high
- 論点: MVPの入力源
- コメント: 最初の入力を GitHub PR に絞っているのは妥当。入力ソースが明確で、実装と説明の両方で有利
- 残す価値: スコープ管理に効くので維持すべき

### R-003
- 優先度: medium
- 論点: 出力フォーマット
- コメント: `schema_version` を持つ構造化出力前提にしているのは強い。後から項目を増やす設計として健全
- 残す価値: LLM出力を契約として扱う方針は維持すべき

### R-004
- 優先度: medium
- 論点: provider 抽象化
- コメント: Ollama を第一候補にしつつクラウドLLMへ切替可能にしているのは現実的
- 残す価値: 品質とコストの両面で柔軟性がある

---

## 先に詰めるべき点

### R-101
- 優先度: critical
- 論点: MVPがまだ少し広い
- コメント: 現状のMVPには `GitHub App導入` `Webhook受信` `PR解析` `学び抽出` `週報生成` `UI閲覧` が入っている。これは最初の1サイクルとしては広い
- 懸念: 基盤実装で時間を使い、肝心の「学びが残るか」の検証が遅れる
- 推奨アクション: 最小の検証単位を `1つのPRを手動投入して learning_items を出せる` まで落とした `MVP-0` を別に定義する

### R-102
- 優先度: critical
- 論点: 学び抽出の良し悪しをどう判定するか
- コメント: `学び3件以上を安定して抽出できる` は量の条件であって、質の条件ではない
- 懸念: それっぽいが使えない出力でも前進したように見えてしまう
- 推奨アクション: 評価基準を追加する
- 評価観点例:
  - 再利用可能か
  - evidence に結びついているか
  - 次回の行動に変換されているか
  - 単なる要約で終わっていないか

### R-103
- 優先度: medium
- 論点: GitHub App導入の現実的な重さ
- コメント: GitHub App は正しい選定であり、LT時点で完成必須ではない前提なら大きな弱点ではない
- 懸念: 初速は落ちるが、完成系の設計としては妥当
- 推奨アクション: 実装フェーズを2段階に分ける
- 具体案:
  - Phase 0: ローカルのサンプルPR JSON で解析パイプラインを作る
  - Phase 1: GitHub App と Webhook を接続する

### R-104
- 優先度: high
- 論点: 前処理の責務が大きい
- コメント: 精度の大半は LLM 本体より、レビューコメントの粒度調整と evidence の整形で決まりそう
- 懸念: ここが曖昧なままだとモデル比較以前に出力がぶれる
- 推奨アクション: 前処理の責務を先に明文化する
- 前処理候補:
  - bot コメント除去
  - 重複指摘の統合
  - 軽微な typo 指摘の扱いルール
  - 指摘と修正差分の対応付け

### R-105
- 優先度: medium
- 論点: 週報生成の価値
- コメント: 本当に新規性があるのは `PRから学びを抽出すること` で、週報化はその表示レイヤーに近い
- 懸念: 週報という器に先に引っ張られると、本質の検証が薄まる
- 推奨アクション: 実装順は `学び抽出` を先、`週報生成` を後に固定する

### R-106
- 優先度: high
- 論点: 検索は後回しでよい
- コメント: 検索は魅力的だが、最初の価値検証には必須ではない
- 懸念: 早期に検索まで触ると、保存・抽出・評価の軸がぼやける
- 推奨アクション: 検索は `Phase 5` のままでよい。MVP完了条件からも外してよい

### R-107
- 優先度: high
- 論点: 認証設計
- コメント: 「誰が使うか」が曖昧なまま API と UI の設計が進むと、後から破壊的変更が入りやすい
- 懸念: 個人ツールなのか、他者利用を見据えるのかで設計の前提が変わる
- 推奨アクション: 初期段階は `個人用ツール` と明記し、認証は将来拡張として切り離す

## 実装順の再提案

### Step 1
- サンプルPRデータを固定する
- 解析対象を3件程度用意する

### Step 2
- 前処理だけを実装する
- コメント正規化と evidence 生成を固める

### Step 3
- LLM出力スキーマを固定する
- `learning_items` の品質を見る

### Step 4
- 週報にまとめる
- ここで初めて digest 層を入れる

### Step 5
- GitHub App と webhook を接続する

### Step 6
- UIを整える
- 閲覧導線と再分析導線を載せる

## 実装前に追加したい定義

### 1. learning_item の完成条件
- どの状態なら「使える学び」とみなすか

### 2. evidence の最小要件
- どの粒度で PR コメントや差分と結びつけるか

### 3. ノイズの扱い
- typo
- style only
- reviewer の好み

### 4. 週報の役割
- 単なる一覧か
- 優先順位付きの振り返りか
- 次回チェックリスト生成までやるか
